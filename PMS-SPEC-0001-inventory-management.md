## 1. Background（背景）

- **現状の課題**
  - 部品在庫が Excel・基幹システム・現場メモに分散している
  - 生産計画に対する引当状況がリアルタイムで見えない
  - 不足が工程遅延後に判明し、ライン停止や手戻りが発生する
- **対象ユーザー**
  - 生産管理（計画・全体調整）
  - 資材担当（在庫・発注判断）
  - 現場班長（引当確認・実績連携）
- **期待する成果**
  - 在庫・引当・不足を1画面で把握できる
  - 不足を事前検知し、ライン停止リスクを低減する
  - 日次で在庫推移レポートを自動生成する
- **なぜ今やるか**
  - 90日計画の Phase 2（31〜60日）の中核機能
  - 生産計画・工程進捗 Spec との連携前提として在庫を先に固める

## 2. Scope（対象範囲）



### In Scope

- [x] **在庫**（本 Spec の主対象）
  - 部品在庫の参照・更新
  - 生産計画に対する引当（Allocation）
  - 安全在庫との比較・不足アラート
  - 入出庫履歴の記録
- [ ] 生産計画（参照・引当連携のみ。計画登録は別 Spec）
- [ ] 工程進捗（不足アラート表示連携のみ）
- [ ] 品質（ロット単位の在庫追跡は v1.1 候補）



### Out of Scope

- 自動発注・発注書生成
- サプライヤー EDI / 外部 WMS との双方向同期
- 会計・原価計算
- 倉庫内ロケーション（棚番）の詳細管理



### 前提条件

- 部品マスタ（`part_code`, 名称, 単位, A/B/C 区分）が登録済み
- 安全在庫マスタ（部品ごとの `safety_stock`）が登録済み
- 生産計画 API（日次・ライン別）が参照可能（未実装の場合はモック可）
- 認証: 社内 ID / SSO
- 対象工場/ライン: （記入）渋谷工場/スカイライン

## 3. Business Rules（業務ルール）


| Rule ID | ルール                                   | 根拠/備考   |
| ------- | ------------------------------------- | ------- |
| BR-001  | 現在庫 = 前回残 + 入庫 − 出庫 − 引当済み数量          | 在庫計算の基本 |
| BR-002  | 引当可能数量 = 現在庫 − 既引当数量                  | 二重引当防止  |
| BR-003  | 現在庫 ≤ 安全在庫 のとき「不足」と判定                 | 社内基準    |
| BR-004  | 同一部品の不足アラートは 24 時間以内に 1 回まで（重複抑制）     | 通知過多防止  |
| BR-005  | A 区分部品の不足は即時通知（メール/Teams は Q-001 で決定） | ライン影響大  |
| BR-006  | 在庫更新・引当・手動調整は監査ログに記録                  | 監査要件    |
| BR-007  | マイナス在庫は原則不可（API で拒否。例外は管理者のみ）         | データ整合   |

## 5. Architecture（設計方針）



### 5.1 技術構成

- **Frontend**: TypeScript（React）
- **Backend API**: TypeScript（NestJS または Express）
- **Batch / レポート**: Python（FastAPI + 日次ジョブ）
- **DB**: PostgreSQL
- **通知**（任意）: メール / Teams Webhook



### 5.2 責務分担


| レイヤ        | 担当            | 内容                          |
| ---------- | ------------- | --------------------------- |
| TypeScript | UI / API      | 在庫 CRUD、引当、不足一覧、権限、リアルタイム表示 |
| Python     | Batch         | 日次在庫サマリー、不足傾向分析、レポート出力      |
| 共通         | Spec / Schema | テーブル定義、イベント形式、エラーコード        |




### 5.3 データ影響


| テーブル/エンティティ             | 変更種別               | 概要            |
| ----------------------- | ------------------ | ------------- |
| `parts`                 | Read               | 部品マスタ（既存）     |
| `inventory_items`       | Create/Update/Read | 部品ごとの現在庫・引当済み |
| `inventory_movements`   | Create/Read        | 入出庫履歴         |
| `inventory_allocations` | Create/Update/Read | 計画に対する引当      |
| `inventory_alerts`      | Create/Read        | 不足アラート        |
| `inventory_audit_logs`  | Create/Read        | 監査ログ          |


**inventory_items（例）**


| カラム           | 型         | 説明           |

## 10. Rollout / Operations（展開・運用）



### 10.1 リリース計画


| Phase | 内容                        | 目安      |
| ----- | ------------------------- | ------- |
| MVP   | REQ-001〜005（参照・入出庫・引当・不足） | 31〜45 日 |
| v1.1  | REQ-006〜007（履歴・即時通知）      | 46〜55 日 |
| v1.2  | REQ-008（日次レポート）           | 56〜60 日 |




### 10.2 監視項目

- 不足アラート件数 / 日
- 在庫 API エラー率・レイテンシ
- マイナス在庫発生件数（0 であること）
- 日次バッチ成功/失敗



### 10.3 Runbook（障害時）

1. **在庫が合わない**: movements / allocations の直近更新を確認 → 監査ログで操作者特定
2. **不足アラートが出ない**: 安全在庫マスタ、判定 API ログ、24h 抑制を確認
3. **一覧が遅い**: DB インデックス（part_code, updated_at）、N+1 確認
4. **一時回避**: Excel 在庫表へ切替（手順書リンクを運用チームが保持）



### 10.4 ロールバック

- **条件**: 在庫数の大量不整合、引当二重化
- **手順**: 機能フラグ OFF → 読取専用モード → データ修復スクリプト（別 Runbook）

## 11. Security / Compliance（社内必須）

- **個人情報**: 本ドメインは部品・数量中心（担当者名は監査ログの user_id のみ）
- **権限モデル**
  - 管理者: 全操作 + 手動調整
  - 資材: 入出庫・引当・参照
  - 現場: 参照 + 不足一覧（更新不可）
- **監査ログ対象**: 入出庫、引当、手動在庫調整、不足アラート解消
- **データ保持**: 履歴・監査 90 日（要件変更時は NFR と同期）
